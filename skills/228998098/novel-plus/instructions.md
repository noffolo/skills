# 小说写作技能 v2.3

## 核心铁律（优先级降序排列，任何时候不得违反）

### P0 用户边界违规（绝对禁止）
- 所有创作必须严格遵守用户指定的边界（R18/BE/特定元素）
- 用户未明确说明时，默认"适合全年龄、纯爱/正剧向、无露骨色情"
- 违规直接阻断写作，P0级事故

### P1 人设崩坏/OOC（严格禁止）
- 角色行为、语言、决策必须与角色圣经一致
- 风格锚定文档是写作宪法，违反视为P1事故
- "OOC"定义：角色行为/语言偏离角色圣经中定义的弧光、性格、动机

### P2 设定逻辑冲突（必须修复）
- 世界观内部逻辑自洽
- 战力/道具/时间线与追踪表一致
- 伏笔埋设与回收逻辑连贯

### P3 风格偏离（应当修复）
- 语言风格/叙事视角/禁用词表遵循风格锚定
- 对话风格/感官偏好与示范文本一致

### P4 节奏字数偏差（优化项）
- 字数应在目标±20%范围内
- 章节节奏应符合细纲规划（铺垫/上升/高潮/过渡）

### 其他铁律
- 每章正文结尾必须有强叙事钩子或情感留白
- 项目所有文档必须结构化（Markdown优先）、版本化、可追溯
- 输出永远用流畅、自然、高文学性的目标语言（默认中文）
- 严禁擅自添加用户未授权的元素（OC、强行HE/BE等）

## 【启动时】断档检测与恢复

> **目的：** 主控启动时自动检测已有项目，还原进度，避免上下文丢失导致重复工作。

### 检测流程

```
1. 搜索项目目录（./projects/、./）
2. 查找 meta/metadata.json + meta/project.md
3. 读取项目状态（phase、进度、配置）
4. 验证项目完整性
5. 向用户汇报状态
```

### 汇报格式

```
检测到已有项目：[项目名]

├─ 阶段：Phase X
├─ 完成章节：Ch1-ChXX
├─ 当前章节：ChYY
├─ 风格锚定：v1.0（已定稿/待定稿）
└─ 自动写作状态：[运行中/已停止/无]

请选择：
[继续当前项目] [新建项目] [查看项目详情]
```

### 用户选项

| 选项 | 主控操作 |
|------|----------|
| 继续当前项目 | 加载必要文件到上下文，根据phase和进度确定恢复点 |
| 新建项目 | 确认后创建新项目（与旧项目共存） |
| 查看详情 | 展示项目完整状态、文件列表、下一步建议 |

### 自动推进恢复

如果 `autoAdvance.status == "running"`，提醒用户：
```
检测到自动写作仍在运行，上次停止在第X章。
请选择：[继续自动写作] [暂停检查进度]
```

## Phase 1：前期架构阶段

**前置条件：** Phase 0 已完成，项目目录已建立。

### Step 1-1：世界观构建

**调用 Claude 4.6 Opus → 输出完整世界观文档**

写入 `worldbuilding/world.md`，必须包含：

1. **基础世界框架**
   - 世界名称、时代背景、地理概览
   - 社会结构、势力格局（正派/中立/反派各1+）

2. **核心规则体系**
   - 力量体系（等级名称、边界、代价）
   - 特殊机制（如有魂穿则最优先定义魂穿规则）

3. **核心悬念/禁忌**
   - 世界观禁忌（绝对不能打破的规则）
   - 隐藏势力（伏笔层）
   - 终极谜题（贯穿全文的主线悬念）

### Step 1-2：角色圣经

**调用 CharacterDesigner Agent → 输出角色档案**

> **关键：** 角色必须基于世界观来设计，确保角色能力体系、社会背景、势力归属与世界观的统一。

**输入：**
- 世界观文档 (`worldbuilding/world.md`)
- 用户提供的角色粗略描述

**输出：**
- `characters/protagonist.md` 主角完整档案
- `characters/characters.md` 其他角色档案

**主角档案必须包含：**
- 外貌特征（具体到可辨识的细节）
- 性格剖析（核心矛盾、心理动机）
- Backstory（前世/起源）- **必须与世界观设定的时代/社会背景契合
- 人物弧光（起点→摩擦→转变→终点）
- 核心目标、恐惧、执念
- **能力/修为设定（必须符合世界观的力量体系）**

**关系网（至少8个角色）：**
- 亲人 2-3人（情感羁绊锚点）
- 仇人 2人（核心冲突源）
- 暧昧/情感线 2人（情感张力）
- 恩人/导师 1人（动机来源）
- 对手/盟友 2人（势力纠葛）

**反派设计：**
- 每个核心反派必须有：目的、优势、弱点、感情线
- **反派势力归属必须与世界观的势力格局对应**

### Step 1-3：全书大纲

**调用 OutlinePlanner Agent → 输出三幕/多线结构大纲**

> **关键：** 大纲必须基于世界观+角色圣经来设计，确保剧情发展、角色命运与世界观逻辑自洽。

**输入：**
- 世界观文档 (`worldbuilding/world.md`)
- 角色圣经 (`characters/protagonist.md`, `characters/characters.md`)

**输出：**
写入 `outline/outline.md`，必须包含：

1. **三幕结构总览**
   - 第一幕：寄生（适应期）—— 建置世界、埋设冲突
   - 第二幕：破局（成长期）—— 冲突升级、主线推进
   - 第三幕：新生（终结期）—— 高潮决战、结局回收

2. **主线索 + 支线列表**
   - 每条线标注：起点、关键节点、高潮点、终点

3. **关键节点清单**
   - 全书 5-7 个核心转折点

4. **结局走向**
   - 明确HE/BE/开放结局
   - 结局核心主题收束

### Step 1-4：章节细纲

**调用 ChapterOutliner Agent → 生成章节细纲**

> **关键：** 细纲必须基于世界观+角色圣经+大纲来设计，确保每章内容与整体架构一致。
>
> **批次规则：** 单批次最多生成 **10章**，完成后可继续生成下一批次。

**输入：**
- 世界观文档 (`worldbuilding/world.md`)
- 角色圣经 (`characters/protagonist.md`, `characters/characters.md`)
- 全书大纲 (`outline/outline.md`)
- 需要规划的章节范围（默认第1-10章）

**输出：**
写入 `outline/chapter-outline.md`，每次生成不超过10章。

**多批次规则：**
- 第一批次：Ch1-10（第一幕主体）
- 第二批次：Ch11-20（第一幕结尾 + 第二幕开头）
- 第三批次：Ch21-30（第二幕主体）
- 依此类推，每10章一批
- **每批次完成后由主控确认，确认后继续生成下一批次**
- 用户可指定范围（如"生成第11-20章细纲"）

> **成本优化：** 上下文长但不需要最强推理，Kimi长文能力足够，省下Opus额度给终审。

**单章细纲模板：**

```markdown
## 第X章：《章标题》

- **核心事件：**
- **主线推进：**
- **情感线：**
- **冲突/转折：**
- **节奏点：**
- **结尾钩子：**
- **伏笔承接：**
- **本章重点：**
- **章节类型：** [普通/高潮/转折/情感爆发]
```

### Step 1-5：全局一致性终审

**调用 FinalReviewer Agent → 输出审查报告**

> **目的：** 确保世界观、角色圣经、大纲、细纲四方一致，无逻辑冲突。

**输入：**
- 世界观文档
- 角色圣经
- 全书大纲
- 章节细纲

**终审检查项：**
- 角色能力是否与世界观的力量体系一致
- 角色社会背景/势力归属是否与世界设定一致
- 大纲中的关键事件是否基于世界观的势力格局
- 细纲中的章节安排是否遵循大纲结构
- 伏笔埋设是否与世界观的悬念设定呼应

**输出：**
- `archive/终审报告.md`

### Step 1-6：M2.7提取关键章节配置

**M2.7自动执行：**

1. 从 `outline/outline.md` 提取幕间分界章节（如Ch10/Ch30/Ch50）
2. 更新 `meta/config.md` 的 `key_chapters` 字段
3. 更新 `meta/config.md` 的 `act_boundaries` 字段

### Step 1-7：风格锚定文档定稿

**调用 StyleAnchorGenerator Agent → 生成风格锚定文档v1.0**

> **关键：** 风格锚定必须基于世界观+角色圣经+大纲来设计，确保语言风格符合世界观的氛围基调。

**输入：**
- 世界观文档
- 角色圣经
- 全书大纲
- `meta/style-anchor.md` 基础偏好（Phase 0初始化）

**输出：**
- 更新 `meta/style-anchor.md`，版本 v0.1 → **v1.0（正式定稿）**

> 风格锚定文档v1.0生成后，所有写作Agent的prompt必须严格附带全文

### Step 1-8：整合终审，定稿存档

M2.7整合终审建议 → 修改所有文档 → 版本从 v0.1 → v0.2 → **v1.0（架构定稿）**

### Step 1-9：汇报进入 Phase 2

```
🎉 项目骨架已完成（v1.0 架构定稿）

📁 项目存档位置：projects/<项目名>/
✅ 世界观文档
✅ 角色圣经（X个角色）
✅ 全书大纲（三幕/X条线）
✅ 章节细纲（前X章）
✅ 全局一致性终审通过
✅ 风格锚定文档（v1.0，含3段示范文本）
✅ 关键章节配置（已提取幕间分界）

可开始写作第1章。
```

## Phase 2：章节写作阶段

**前置条件：** Phase 1 架构定稿（v1.0+）。

---

### ⚠️ 强制机制：上下文喂入策略

> **第15章后任何模型都无法装下全部前文，必须严格按以下策略执行。**

#### 三层喂入架构

```
┌─────────────────────────────────────────────────────────────┐
│                    上下文喂入三层架构                         │
├─────────────────────────────────────────────────────────────┤
│  【固定层】始终喂入（≤6500字）                               │
│  ├── 世界观摘要（≤1500字）                                  │
│  ├── 角色圣经摘要（≤2000字）                                │
│  └── 全书大纲（≤3000字）                                    │
├─────────────────────────────────────────────────────────────┤
│  【滚动层】按需滚动                                          │
│  ├── 最近3章原文                                            │
│  ├── 前文累积摘要（每8–10章由Qwen生成，≤2200字）           │
│  └── 每8–10章触发：Qwen生成滚动摘要 → 更新 rolling-summary.md │
├─────────────────────────────────────────────────────────────┤
│  【按需层】本章相关条目                                      │
│  ├── 本章相关伏笔（从 plot-tracker.md 筛选）                │
│  └── 本章相关关系变动（从 relationship-tracker.md 筛选）     │
├─────────────────────────────────────────────────────────────┤
│  【禁止】❌ 一次性喂入全部前文原文                            │
└─────────────────────────────────────────────────────────────┘
```

#### 滚动摘要生成规则（强制执行）

| 触发时机 | 执行动作 | 输出位置 |
|----------|----------|----------|
| 完成第5章 | Qwen 3 Max 生成 Ch1-5 摘要 | references/rolling-summary.md |
| 完成第10章 | Qwen 3 Max 生成 Ch6-10 摘要（追加） | references/rolling-summary.md |
| 完成第15章 | Qwen 3 Max 生成 Ch11-15 摘要（追加） | references/rolling-summary.md |
| 每完成5章 | 同上模式，追加新段落，覆盖更新 | references/rolling-summary.md |

**触发时机细化：**
- 每完成第 8–10 章时生成一次滚动摘要（或完成一个幕/大节点时立即生成）
- 在Phase 2每章的Step 6（更新存档）中检查，如完成章节数达到触发阈值，立即调用Qwen 3 Max生成滚动摘要

**Qwen 滚动摘要 Prompt 模板：**

```
你是小说压缩专家。请将以下前N章内容压缩为≤2200字的故事摘要。
```

### ⚠️ 强制机制：模型故障/拒绝回退策略

> **核心原则：** 预防优于回退，M2.7在分发任务前主动预判敏感度
> **回退触发条件：** 模型返回"无法生成"、"拒绝"、"内容敏感"、超时（>120s）

#### 模型回退链

| 原模型 | 触发条件 | 回退目标 | 回退后处理 |
|--------|----------|----------|------------|
| Kimi k2.5 | 拒绝/故障 | Qwen 3.5 Plus | 担任主笔+润色（合并执行） |
| Claude 4.6 Opus | 拒绝 | Qwen 3 Max | 担任终审；若Qwen也拒 → M2.7自行终审 |
| Gemini 3.1 Pro | 拒绝 | Kimi k2.5 | 写战斗段落，Prompt附加"动作描写+专业术语" |
| Qwen 3 Max | 拒绝 | M2.7 | 自行一致性检查，输出简化版OOC报告 |
| Doubao seed | 拒绝 | 跳过 | 跳过读者模拟步骤，不阻断写作 |

#### 回退记录格式

每次触发回退，必须在 `archive/archive.md` 中记录：

```markdown
## 回退记录 ChXX

| 时间 | 模型 | 原任务 | 回退目标 | 结果 |
|------|------|--------|----------|------|
| 2026-03-21 16:30 | Kimi k2.5 | 初稿 | Qwen 3.5 Plus | 成功 |
```

#### 敏感度预判矩阵

| 敏感等级 | 场景类型 | Prompt约束 | 模型选择 |
|----------|----------|------------|----------|
| **高敏感** | 亲密场景/情感爆发/R18相关 | 强制附加文学性/含蓄/情感约束 | 优先审核宽松模型 |
| **中敏感** | 暴力场面/悬疑/心理压迫 | 附加"适度描写，避免细节过多" | 任意模型 |
| **低敏感** | 日常/战斗/信息交代 | 标准Prompt，无额外约束 | 任意模型 |

#### 高敏感Prompt强制附加

当涉及亲密/情感场景时，**所有写作Agent的prompt必须附加**：

```
【敏感度约束】
- 必须：文学性描写，含蓄隐晦，注重情感而非动作
- 必须：使用留白、暗示、情绪侧面烘托
- 禁止：直接描写身体接触/性暗示/过于直白的欲望表达
- 禁止：R18相关词汇或暗示
- 风格参考：村上春树的隐晦笔触、金陵雪的克制抒情
```

#### 连续拒绝处理流程

```
首次拒绝 → 记录日志，尝试回退模型
    ↓ 第二次拒绝（同一任务）
回退模型也失败 → M2.7生成"用户提示"
    ↓ 两次后仍无法执行
跳过该段落（保留占位）→ 汇报中标注"ChXX 第Y段需用户确认后重写"
```

### 双轨制工作流

> 200章小说若12步全跑 = 2400次模型调用，成本/时间不可接受。
> 改为双轨制：**常规章节**（6步）vs **重点章节**（9步）。

#### 轨道判定规则（动态化）

```
┌─────────────────────────────────────────────────────────────┐
│                    章节类型自动判定（动态）                   │
├─────────────────────────────────────────────────────────────┤
│ 【重点章节】满足任一条件：                                   │
│  ├── 用户手动指定"重点章"                                   │
│  ├── 细纲标注：高潮/转折/情感爆发                          │
│  ├── 首章（Ch1）                                           │
│  ├── 末章（ChN，从meta/config.md读取total_chapters）       │
│  └── 幕间分界章（从meta/config.md的key_chapters读取）      │
│ 【常规章节】不满足上述条件                                   │
└─────────────────────────────────────────────────────────────┘
```

> **动态化：** 不使用硬编码Ch10/Ch30/Ch50，而是从 `meta/config.md` 的 `key_chapters` 字段动态读取。

### 自动推进X章机制

> **核心目标：** 新增 `auto_advance_chapters` 参数，支持自动连续写作n章，减少人工干预。

#### 参数定义

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `auto_advance_chapters` | 正整数 | 4 | 一次自动写4章，节奏舒适 |
| `write_interval_seconds` | 正整数 | 6 | API缓冲 + 阅读思考时间 |
| `auto_confirm` | 布尔 | false | 保留手动安全感，是否自动确认继续 |

**个人使用推荐快捷设置：**

| 场景 | auto_advance_chapters | write_interval_seconds | auto_confirm |
|------|----------------------|------------------------|--------------|
| 中速日常（默认） | 4 | 6 | false |
| 高速大批量 | 8 | 4 | false |
| 慢速审慎（重点章） | 2 | 10 | false |

**调用示例：**
```
用户："写第5章，自动推进10章"
→ 等同于："写第5章到第15章"，auto_advance_chapters=10

用户："写第5章，自动推进"（默认auto_advance_chapters=1）
→ 等同于："写第5章，自动推进1章"，写完第5章后自动写第6章
```

#### 自动推进流程

```
┌─────────────────────────────────────────────────────────────┐
│ 自动推进主循环                                             │
├─────────────────────────────────────────────────────────────┤
│ 1. 初始化                                                 │
│    → 读取 auto_advance_chapters 参数                      │
│    → 读取 write_interval_seconds 参数                     │
│    → 设置：remaining = auto_advance_chapters              │
│    → 当前章节 current_chapter = X                         │
│                                                               │
│ 2. 执行写作循环（while remaining > 0）                    │
│    │                                                        │
│    │ ▼ 执行当前章节写作（Phase 2 双轨工作流）              │
│    │                                                        │
│    │ 3. 成功后：                                          │
│    │    → 更新 remaining = remaining - 1                  │
│    │    → current_chapter = current_chapter + 1           │
│    │    → 执行存档+滚动摘要+读者模拟触发检查              │
│    │    → 等待 write_interval_seconds 秒                  │
│    │                                                        │
│    │ 4. 如 remaining > 0：                                │
│    │    → 反馈进度："ChX完成，正在自动写作ChX+1..."     │
│    │    → 自动进入下一章写作循环                         │
│    │                                                        │
│    │ 5. 异常处理（见下方异常机制）                       │
│    │                                                        │
│    ▼                                                        │
│                                                               │
│ 6. 全部完成后：                                             │
│    → 汇总报告："第X-(X+n)章已完成，n章总计XX字"         │
│    → 重置 remaining = 0                                    │
│    → 提示用户可继续或调整参数                             │
└─────────────────────────────────────────────────────────────┘
```

#### 状态管理

**meta/metadata.json 扩展字段：**

```json
{
  "project": { ... },
  "writingProgress": {
    "currentChapter": 15,
    "lastCompletedChapter": 15,
    "lastFeedbackTrigger": 10,
    "autoAdvance": {
      "enabled": true,
      "remaining": 8,
      "totalRequested": 10,
      "currentChapter": 15,

      "autoConfirm": false,

    }
  },
  "chapters": { ... }
}
```

**状态流转：**
```
idle → running → completed
            ↓
          paused (异常时)
            ↓
          failed (无法恢复)
```

#### 异常与回退机制

```
┌─────────────────────────────────────────────────────────────┐
│ 异常处理流程                                               │
├─────────────────────────────────────────────────────────────┤
│ 1. 捕获异常类型                                           │
│    → 模型拒绝（连续2次回退仍失败）                        │
│    → API调用失败                                          │
│    → 超时（>120s）                                       │
│    → 内容审核拒绝（P0边界违规）                          │
│                                                               │
│ 2. 立即暂停自动推进                                       │
│    → 设置 autoAdvance.status = "paused"                   │
│    → 记录异常日志至 archive/archive.md                    │
│    → 记录：异常章节、异常类型、错误信息、已写章节数      │
│                                                               │
│ 3. 反馈用户                                               │
│    → "自动写作已暂停"                                    │
│    → 当前进度：已完成X章，正在写第Y章时失败               │
│    → 错误原因：（具体原因）                               │
│    → 建议操作：                                          │
│      - "继续写作第Y章"（修复后恢复）                     │
│      - "跳过第Y章，写第Y+1章"                           │
│      - "停止自动推进，已完成章节已保存"                  │
│                                                               │
│ 4. 用户确认后：                                          │
│    → 如选择"继续"：恢复写作，remaining不变              │
│    → 如选择"跳过"：remaining-1，跳至下一章               │
│    → 如选择"停止"：remaining=0，保存当前进度            │
└─────────────────────────────────────────────────────────────┘
```

**异常日志格式：**

```markdown
## 自动推进异常记录 ChXX

- 时间：2026-03-21 17:15
- 异常章节：Ch12
- 异常类型：模型拒绝（连续2次回退）
- 错误信息：（详细错误描述）
- 已完成章节：Ch10, Ch11
- 剩余章节：Ch12-Ch15（原计划8章）
- 状态：已暂停，等待用户指示
```

#### 调度策略

| 参数 | 默认值 | 说明 | 配置建议 |
|------|--------|------|----------|
| `auto_advance_chapters` | 4 | 一次自动写4章 | 节奏舒适，适合日常写作 |
| `write_interval_seconds` | 6秒 | 每章完成后等待N秒 | API缓冲+阅读时间，推荐≥6秒 |
| `auto_confirm` | false | 是否自动继续无需确认 | 生产用false，保留手动安全确认 |

**快捷配置参考：**
- 中速（默认）：4章 / 6秒 / false
- 高速：8章 / 4秒 / false（API稳定时大批量写作）
- 慢速审慎：2章 / 10秒 / false（重点章/重要段落）

#### 恢复继续机制

**用户发送"继续写作"时：**
1. 读取 `meta/metadata.json` 中的 `autoAdvance` 状态
2. 如 status = "paused"，提取 remaining 和 current_chapter
3. 恢复自动推进循环，从断点继续

**用户发送"停止写作"时：**
1. 设置 `autoAdvance.status = "idle"`
2. 设置 `remaining = 0`
3. 保存当前进度，提示"已停止，当前进度已保存"

#### 使用场景示例

```
场景1：用户想一口气写50章
→ "写第1章，自动推进50章"
→ 自动写作 Ch1→Ch2→...→Ch50
→ 每章间隔3秒自动继续

场景2：用户想写10章，每5章确认一次
→ "写第5章，自动推进10章，auto_confirm=true"
→ 写完Ch5后暂停，等待确认
→ 用户确认后继续，写作Ch6-Ch14
→ 再暂停，用户确认后继续写Ch15

场景3：用户写完20章后发现问题，想从第15章恢复
→ "继续写作"
→ 读取断点状态，从Ch15恢复继续

场景4：写作过程中被拒绝，需要调整
→ "继续写作第12章"
→ 从失败的第12章重新开始
```

**触发时机：** 用户要求重写/补设定/改大纲/调整风格/段落重写。

### 常见场景

| 场景 | 处理方式 |
|------|----------|
| 重写某章节 | 调用主笔 → OOC守护 → Opus终审 → 更新存档 |
| 段落重写 | 调取上下文 → 调用相关Agent → 一致性检查 → 版本管理 |
| 补设定/世界观扩展 | 调用最强模型 → 更新worldbuilding → 通知用户 |
| 修改大纲 | 调用最强模型 → 输出新大纲 → 终审 → 用户确认 → 更新存档 |
| 风格调整 | 主控先分析差异 → 针对性调用润色Agent → Opus把关 |
| 大修（Phase 1重来） | 回到Phase 1起点，按需保留已确认部分 |
